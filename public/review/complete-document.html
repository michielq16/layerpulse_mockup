Complete Document — the canonical full-scope model write-up

One document per semantic model that contains everything. The 4 audience presets (Auditor / Analyst / Executive / Engineer) become lenses over it — same source, four subsets, zero content drift.

In one minute

What this is

When a tenant needs the definitive write-up of a semantic model — for certification, for handing over to a new partner, for an auditor evidence baseline — the Complete document is it. It includes every section in the master catalogue, is tiered for signal density, and ends with a deterministic KEEP / OPTIMIZE / RETIRE verdict on the cover.

Why a separate "Complete" document, on top of the 4 audience presets?

The audience presets answer "what does this reader need?". Complete answers "what is the full-scope canonical write-up?". Without a canonical base, four parallel renderers drift apart over time — a description in the auditor doc disagrees with the engineer doc. Solve it by making Complete the source and the presets filters over it. One render, four lenses.

Locked principles

The trust contract behind every render

Glossary-driven descriptions.
Measure / column / table descriptions come ONLY from attached business-glossary terms. No term attached → blank. Never invented prose. The document grows with the glossary, not faster.
Source tagging on every group.
Auto · Manual · Derived chips at the top of every section. The reader always knows what's machine-extracted vs human-captured vs computed.
Ownership → cover + sign-off.
Owner / SME / Stewards / Domain flow from /ownership into the cover credit and the back-page sign-off. Missing → explicit empty-state, never fabricated.
Degradation, never fabrication.
Automated sections always render. Manual sections render explicit empty-states ("No stewards assigned — add via /ownership") — the empty slot is the prompt to populate the foundation.
Completeness meter on the cover.
Five chips showing the human-curated coverage at a glance — Owner / SME / Stewards / Glossary / Processes.
The verdict logic

KEEP / OPTIMIZE / RETIRE — deterministic, no AI

A single line on the cover that the reader walks away with. The rest of the doc is the justification.

axes:    Adoption · Cost · Quality · Refresh
rag:     g (good) | a (amber) | r (red)
verdict: RETIRE   if adoption.rag === 'r'
         OPTIMIZE if any red OR ≥2 amber
         KEEP     otherwise

Thresholds: Adoption green if MAU ≥ 100 & dormancy < 40%; Cost green if share-of-capacity ≤ 20%; Quality green if score ≥ 8; Refresh green if last-90d ok-rate ≥ 95%. Amber/red bands one step softer each.
